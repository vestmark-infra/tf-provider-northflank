# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Toolchain

Go is managed by [mise](https://mise.jdx.dev/) — it is not on the system PATH by default. Always prefix `go` commands with `~/.local/bin/mise exec --`:

```sh
~/.local/bin/mise exec -- go build ./...
~/.local/bin/mise exec -- go test ./...
```

`make` targets handle this automatically. Go 1.25, oapi-codegen v2, and tfplugindocs are all pinned in `mise.toml`.

## Common commands

```sh
make build       # compile the provider binary
make install     # install binary to $GOBIN (needed before local terraform testing)
make test        # unit tests (no network, no Northflank account needed)
make testacc     # acceptance tests — requires TF_ACC=1, NORTHFLANK_API_TOKEN, NORTHFLANK_TEST_PROJECT
make generate    # re-run oapi-codegen after changing openapi/northflank-openapi.json
make fmt         # gofmt -s -w .
make vet         # go vet ./...
make tidy        # go mod tidy
```

Run a single test package or test function:

```sh
~/.local/bin/mise exec -- go test ./internal/client/... -run TestGetProjectSecret_NotFound -v
```

## Local terraform testing

Install the binary, then add dev_overrides to `~/.terraformrc`:

```hcl
provider_installation {
  dev_overrides { "vestmark-infra/northflank" = "/home/kbrooks/.local/share/mise/installs/go/1.25.11/bin" }
  direct {}
}
```

Do **not** run `terraform init` with dev_overrides active — go straight to `terraform plan`/`apply`.

## Architecture

The provider has three layers, each insulating the next from churn:

```
internal/nfapi/       ← generated (DO NOT EDIT)
internal/client/      ← hand-written wrapper
internal/provider/    ← Terraform Plugin Framework
```

**`internal/nfapi/northflank.gen.go`** — auto-generated by oapi-codegen from `openapi/northflank-openapi.json`. Contains typed request/response structs and a `ClientWithResponses` for each included operation. Type names are long and inline (e.g. `PostProjectsSecrets200JSONResponseBodyDataSecretType`) because the Northflank spec has no shared component schemas. Never edit this file manually.

**`internal/client/`** — wraps `nfapi.ClientWithResponses` with:
- Bearer auth injected via `WithRequestEditorFn` in `client.go`
- `ErrNotFound` sentinel (used by resource `Read` to call `RemoveResource`)
- Stable, flat model types (`SecretGroup`, `Team`, `Project`) in `secrets.go` / `teams.go` that decouple the provider from generated symbol names
- Paginated list methods (`ListTeams`, `ListProjects`, `ListTeamProjects`) that exhaust all cursor pages

**`internal/provider/`** — uses `terraform-plugin-framework` (protocol v6). The provider's `Configure` sets `resp.ResourceData = client` and `resp.DataSourceData = client`; each resource/data source retrieves it in their own `Configure` via `req.ProviderData.(*client.Client)`. If `api_token` or `base_url` is unknown during plan (e.g. sourced from a data source), `Configure` returns early without building the client — Terraform re-calls it at apply time with resolved values.

**Adding a new resource or data source:**
1. Add the relevant `operationId`(s) to `openapi/oapi-codegen.yaml` and run `make generate`
2. Add typed methods to `internal/client/` using the generated `nfapi` types
3. Create `internal/provider/<name>_resource.go` or `<name>_data_source.go` following the existing pattern
4. Register the constructor in `provider.go` `Resources()` or `DataSources()`

## OpenAPI spec pipeline

The Northflank spec at `https://api.northflank.com/v1/swagger-json` is a hybrid Swagger-2/OpenAPI-3 document and cannot be fed to oapi-codegen directly. The normalization pipeline:

1. `openapi/fetch.sh` — downloads the raw spec to `openapi/northflank-raw.json` (gitignored)
2. `openapi/normalize.py` — fixes the hybrid spec:
   - Resolves method-level `$ref`s into `components.pathItems` (where all operations actually live)
   - Rewrites `response.schema` → `response.content["application/json"].schema` (598 instances)
   - Strips Swagger-2 cruft from nested param `schema` objects
   - Converts array `examples` to OA3 map form
   - Drops `/v1/teams/{teamId}/...` mirrors (identical operationIds cause duplicate generated types), **except** `/v1/teams` and `/v1/teams/{teamId}/projects` which get unique operationIds (`getTeams`, `getTeamProjects`)
   - Injects `bearerAuth` securityScheme
3. The normalized output `openapi/northflank-openapi.json` is committed and used as the codegen input

Re-run the full pipeline with: `bash openapi/fetch.sh && make generate`

## Key design decisions

- **`project_id` requires replace** on `northflank_secret` — secrets are project-scoped and cannot be moved
- **Update uses PATCH** (not PUT) — only changed fields are sent
- **`variables` is Sensitive** — masked in plan output; the GET endpoint returns decrypted values so import works cleanly
- **Import format** for `northflank_secret`: `<project_id>/<secret_id>`
- **`northflank_project` data source** uses `team_id` to call `/v1/teams/{teamId}/projects` when set; otherwise falls back to `/v1/projects` (token-implicit team). Both paginate fully before filtering by name.
- **Multi-team pattern**: `northflank_team` (name → id) → `northflank_project` (name + team_id → id) → `northflank_secret` (project_id)
